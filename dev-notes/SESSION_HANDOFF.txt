================================================================================
ARCHITECTURE DECISION: WHY ONE VESTING CONTRACT, NOT MULTIPLE SALE CONTRACTS
================================================================================

The industry standard (Uniswap, Aave, Compound, most serious seed rounds) is:
  - One shared Vesting contract with per-beneficiary, per-schedule terms.
  - One Sale contract per pricing tranche (if price changes between rounds).
  - OTC / negotiated terms handled via mintOTC() with custom cliff/duration.

Multiple Sale contracts for different vesting terms is an anti-pattern:
  - Multiplies audit surface and deployment cost.
  - Admin overhead scales with number of contracts.
  - The differentiation is purely schedule parameters — already per-address data.

Current setup supports:
  - Standard public sale: buy() with global defaultCliff/defaultDuration.
  - Seed/OTC bespoke terms: mintOTC(recipient, amount, cliff, duration).
  - Top-up allocations: second mintOTC to same address creates schedule ID 1, 2...
  - Different terms per investor: each mintOTC call specifies its own cliff/duration.


================================================================================
INVESTOR VESTING DASHBOARD
================================================================================

All data is on-chain and readable at zero gas cost. The dApp should:

  1. Call scheduleCount(investorWallet) to get total number of schedules.
  2. For each scheduleId from 0 to count-1, call:
       getVestingSchedule(investorWallet, scheduleId)
     Returns: totalAmount, released, startTime, cliff, duration, revoked, releasable
  3. Alternatively call getAllSchedules(investorWallet) for the full array in one call.
  4. Display per schedule:
       - Total allocation
       - Cliff end date (startTime + cliff as unix timestamp)
       - Vesting end date (startTime + duration)
       - % vested so far
       - Claimable now (releasable)
       - Claimed to date (released)
  5. Release button calls release(scheduleId) on the Vesting contract.

The Vesting contract address must be stored in the dApp config. It is set on the
Sale contract via setVestingConfig() and is readable as sale.vestingContract().

For OTC deals (off-chain payment, bespoke terms):
  sale.mintOTC(investorWallet, fvcAmount, cliffSeconds, durationSeconds)
  e.g. 12mo cliff, 24mo vest: sale.mintOTC(wallet, 4000000e18, 365 days, 730 days)
  e.g. no vesting (direct):   sale.mintOTC(wallet, 1000000e18, 0, 0)
